### Reflow와 Repaint의 차이는 무엇인가요?

Reflow는 DOM 요소의 크기와 위치가 변경될 때 브라우저에서 레이아웃을 다시 계산하는 과정으로,
초기 렌더링 뿐만 아니라 요소의 크기 변경, DOM 추가/삭제, 폰트 변경 등에서도 발생합니다.

Repaint의 경우에는 레이아웃 변화 없이 색상이나 배경등 시각적 요소만 변경될 때 발생하며,
실제 픽셀을 다시 그리는 과정입니다.

Reflow는 레이아웃을 계산하는 과정이 포함되기 때문에 비용이 크고, Repaint보다 성능에 더 큰 영향을 줍니다.

### 🎃꼬리질문

#### 그렇다면 Reflow를 최소화하는 방법은 무엇인가요?

Reflow를 최소화하기 위해서는 DOM의 조작을 최소화하고, 여러번 변경해야 하는 작업을 한번에 처리하는 것이 중요합니다.
또한 dispaly:none 과 같이 상태에서 요소를 수정하거나, DocumentFragement를 활용해 메모리에서 작업 후에 DOM에 한번만 추가하는 방법이 있습니다.
그리고 레이아웃 값을 읽는 작업과 DOM 변경 작업을 분리해서 불필요한 Reflow발생을 방지할 수 있습니다.



#### 브라우저에서 레이아웃을 강제로 발생시키는 속성에는 무엇이 있나요? (어떤 코드를 쓰면 브라우저가 강제로 layout계산을 하게 되는지)

offsetWidth, offsetHeight,clientWidth ,getBoundingClientRect와 같이 **레이아웃 정보를 조회하는 속성은 브라우저가 최신 값을 보장하기 위해서 강제로 Reflow를 발생**시킵니다.



#### React에서 이런 Reflow문제를 어떻게 줄이나요?

React는 Virtual DOM을 사용하여 이전 상태와 변경된 상태를 비교하는 Diffing과정을 거친 후, 실제 DOM에 필요한 변경만 적용합니다.
또한 여러 상태 변경을 한번에 모아서 처리하는 Batching을 통해 DOM 업데이트 횟수를 줄입니다.
이로 인해 불필요한 DOM조작과 Reflow 발생을 최소화 할 수 있습니다.
하지만 Virtual DOM도 결국 실제 DOM 업데이트가 발생하기 때문에 Reflow 자체를 없애는 것이 아니라 최소화하는 방식으로 볼수 있습니다.



#### key를 사용하는 이유는 무엇인가요?

Diffing -> 같은 요소인지 판단하기 위해서 Key를 사용


React에서 Key는 리스트 렌더링 시 각 요소를 고유하게 식별하기 위한 값으로,
Diffing과정에서 이전 요소와 현재 요소를 비교할 때 Key를 기준으로 동일한 요소인지 판단하여, 
불필요한 DOM 변경을 줄이고 성능을 최적화하기 위해서 사용됩니다.



#### key로 index를 사용하면 안되는 이유는?

index key = 위치 기준 -> 요소 identity 깨짐 -> 버그 발생

index를 key로 사용할 경우, 리스트의 순서가 변경되거나 요소가 추가/삭제될 때 각 요소의 고유성이 유지되지 않아 DOM이 재사용될 수 있습니다.
이로 인해 state가 꼬이거나 불필요한 렌더링이 발생할 수 있기 때문에 key는 변하지 않는 고유한 값을 사용하는 것이 중요합니다.



#### useEffect의 실행 타이밍에 대해서 설명해주세요.

렌더링 -> 화면 반영 -> useEffect 실행 ( 화면 그린 다음 실행 )

useEffect는 컴포넌트가 렌더링된 이후에 실행되는 Hook입니다. dependency array에 따라 실행 시점이 달라지며, 
값이 없으면 렌더링마다 실행되고, 빈 배열일 경우 마운트시 한 번만 실행되며, 
특정 값이 있을 경우 해당 값이 변경될 때 실행됩니다.



#### useEffect와 useLayoutEffect의 차이는 무엇인가요?

- useEffect -> 화면 그린 후 실행
- useLayoutEffect -> 화면 그리기 전에 실행

useEffect는 렌더링 이후에 실행되어 화면을 그린 후 비동기적으로 동작하며,
useLayoutEffect는 렌더링 직후, 브라우저가 화면을 그리기 전에 동기적으로 실행됩니다.
따라서 useLayoutEffect는 레이아웃 측정이나 DOM 변경과 같이 화면 깜빡임을 방지해야 하는 경우에 사용됩니다.



#### React에서 상태(state)가 변경되면 왜 바로 반영되지 않나요?

- 비동기 처리(Betching) : 바로 적용하지 않고 모아서 한 번에 처리

React의 state는 즉시 변경되는 것이 아니라 내부적으로 batching되어 여러 상태 변경을 한번에 처리합니다.
따라서 setState를 호출해도 바로 값이 반영되지 않고, 다음 렌더링 사이클에서 업데이트된 값이 반영됩니다.

#### React에서 Props와 State의 차이점은 무엇인가요?

👉 props = 외부에서 받는 값 (읽기 전용)
👉 state = 내부에서 바꾸는 값

props는 부모 컴포넌트에서 자식 컴포넌트로 데이터를 전달하기 위한 읽기 전용 값이며, 자식 컴포넌트에서 직접 수정을 할수가 없습니다.
반면 state는 컴포넌트 내부에서 관리 되는 값으로, 사용자 입력이나 이벤트에 따라 변경될 수 있으며 변경 시 컴포넌트가 리렌더링 됩니다.
React는 이러한 구조를 통해 **단방향 데이터 흐름을 유지**합니다.
